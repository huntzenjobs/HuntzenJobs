Où stocker les données de tracking ?

Première décision clé — elle détermine tout le reste de l'architecture

A

PostHog Cloud + Supabase (hybride)

PostHog gère les clics, sessions, funnels, replays (1M events/mois gratuit)
Supabase stocke les events métier critiques (quota dépassé, feature utilisée)
Redis gère la présence temps réel (qui est là maintenant)

Avantages

  • 0 code pour les clics (auto-capture)
  • Funnel, cohort, session replay inclus
  • Dashboard PostHog pour le marketing
  • Gratuit jusqu'à ~50k users

Inconvénients

  • Dépendance externe (RGPD à gérer)
  • Data dispersée entre 3 endroits
B

100% Supabase maison

Supabase stocke TOUS les events (table user_events)
Redis gère la présence temps réel
Admin HuntZen = seul dashboard, tout centralisé

Avantages

  • Tout dans ton infra, 0 dépendance externe
  • Requêtes SQL libres sur tes données
  • RGPD simplifié (1 seul provider)
  • Visible dans l'admin HuntZen déjà existant

Inconvénients

  • Plus de code à écrire (pas d'auto-capture)
  • Pas de funnel/cohort prêt à l'emploi
  • Volume à gérer (Supabase Pro = OK)
C

PostHog + Supabase Realtime pour l'admin live

PostHog pour TOUT le tracking comportemental
Supabase Realtime Broadcast pour pousser les events live dans l'admin
Redis uniquement pour les compteurs ARQ existants

Avantages

  • Architecture la plus simple à implémenter
  • Supabase Realtime déjà payé et configuré
  • Pas de SSE custom à gérer

Inconvénients

  • 2 dashboards (PostHog + admin HuntZen)
  • Présence temps réel moins précise

💡 Ma recommandation : Option A — PostHog pour l'analytics comportemental, Supabase pour les events métier, Redis pour la présence live. Le meilleur équilibre pour un solo dev.
